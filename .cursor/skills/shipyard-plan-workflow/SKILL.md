# Shipyard 规划与交付工作流（项目 Skill）

## 文档落盘位置（必须）

- **统一目录**：本仓库所有版本规划、需求规格、审阅用计划书，一律放在 **`.cursor/plans/`**（项目根下 `.cursor` 内，与 Cursor 配置共存、可入库共享）。
- **勿用全局计划目录作为唯一来源**：Cursor 可能在本机 `~/.cursor/plans/` 生成计划副本；Shipyard 以 **仓库内 `.cursor/plans/`** 为规范与协作事实来源；若发现重复，应合并后删除全局副本或明确标注「非入库草稿」。
- **推荐命名**（与现有惯例一致）：
  - 需求规格：`shipyard-vX.Y-需求规格.md`
  - 路线图（含 YAML 待办可与 Cursor 对齐）：`shipyard-vX.Y-路线图.plan.md`
  - 专题计划：`{主题}.plan.md`
- **禁止**：把唯一事实来源只放在聊天里或用户本机路径；应写入 `.cursor/plans/` 并提交（除非用户明确只要本地草稿）。

## 写作顺序（审阅通过后）

1. **用户审阅**高层计划书/路线图草案（范围、优先级、不做什么）并确认无误。
2. **Agent 先写需求规格**：在 `.cursor/plans/` 落地 **`…-需求规格.md`**（FR/NFR、验收要点、P0/P1/Stretch、与代码锚点文件路径）。
3. **再写或同步路线图**：`**…-路线图.plan.md**` 与需求规格 **互链**；路线图正文可与需求规格摘要一致，避免两处矛盾。
4. **最后**再进入实现阶段（改代码、迁移、测试）；实现时以需求规格为验收依据。

> 若用户跳过需求规格直接要求写代码：仍应提醒并优先补写简短需求规格，除非用户明确只要原型。

## 职责边界：不是「全部由 Agent 做完」

| 角色 | 适合做的事 |
|------|------------|
| **用户** | 定范围与优先级、说「不做什么」、审阅文档结论、产品/合规取舍、合并 PR 等需人担责的决策。 |
| **Agent** | 起草与修订 `.cursor/plans/` 文档、检索代码锚点、实现、跑 `pnpm`/迁移/测试、修 CI。 |

- **不要假设 Agent 包办一切**：战略取舍、对外承诺、删功能等必须由用户明确；Agent 可给选项与草稿，**不替用户拍板**。
- **不要让用户执行终端命令**：在真实环境中有 shell 权限时，**由 Agent 直接执行** `pnpm`、`prisma migrate`、`git` 等；说明里避免「请你本地执行 …」，改为 Agent 已执行或正在执行。

## Cursor 模式切换（无需逐项征求确认）

- 需要架构权衡、多方案对比、大范围拆任务时：Agent **应主动切换到 Plan 模式**（或等价协作模式）再讨论，**不必先问「要不要切换模式」**。
- 方案清楚、进入改代码与跑命令时：**主动切回 Agent 模式**执行。
- **禁止**：把「切换模式」当成必须由用户手动点的步骤写在回复里让用户操作；模式切换由 Agent 按任务阶段自行发起。

## 与现有 Skill 的协作

- **后端实现**：对齐 [ddd-backend](../ddd-backend/SKILL.md)。
- **前端实现**：对齐 [ddd-frontend](../ddd-frontend/SKILL.md)；涉及 UI 规范时联用 [shipyard-ui-spec](../shipyard-ui-spec/SKILL.md)。

## 自检清单（Agent 交付规划类任务前）

- [ ] 需求规格已在 `.cursor/plans/` 且路线图与之互链（若本版本有路线图）。
- [ ] P0/P1/Stretch 与验收表可对照实现。
- [ ] 未要求用户自行运行安装/迁移/测试命令（除非环境确无权限，再说明原因与可选替代）。
- [ ] 未把模式切换列为「请用户点击」的步骤。
